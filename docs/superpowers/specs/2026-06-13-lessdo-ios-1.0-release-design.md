# LessDo iOS 1.0 Release Design

Date: 2026-06-13

## 1. Goal

Deliver LessDo 1.0 as a production-ready, free App Store release for iPhone
and iPad. The release must provide a reliable local-first task and focus
experience, pass automated and simulator validation, request only necessary
permissions, and contain accurate privacy and review materials.

The minimum supported operating system is iOS and iPadOS 16.

## 2. Release Scope

### Included in 1.0

- Create, edit, complete, restore, and delete tasks.
- Multiple user-created lists and a protected default Inbox.
- Today and overdue task views.
- English and Simplified Chinese smart date and time input.
- Due dates, reminders, notes, subtasks, priorities, and daily, weekly, or
  monthly repeat rules.
- Notification actions to complete a task or snooze it for 10 minutes.
- Grocery lists with fast entry and completed-item grouping.
- Pomodoro, countdown, and count-up focus modes.
- Persistent active focus sessions and focus history.
- Themes, system dark mode, dynamic text, and a large-text preference.
- Optional Face ID or Touch ID application lock.
- Text sharing and the documented `lessdo://` URL scheme.
- English and Simplified Chinese localization, following the system language
  by default with an in-app override.
- A short, skippable first-launch introduction followed by an empty Inbox.
- Offline privacy policy, terms of use, and diagnostic-log export.

### Deferred to 1.1 or Later

- iCloud or CloudKit synchronization.
- Collaborative or link-based shared lists.
- Home Screen and Lock Screen widgets.
- Siri and App Intents.
- Apple Calendar integration.
- Location reminders.
- Continuous or escalating reminders.
- Apple Watch support.
- macOS, Android, and web store releases.

Deferred features must not appear as disabled switches, simulated behavior, or
promised paid functionality in the 1.0 user interface.

### Commercial Model

LessDo 1.0 is free. It contains no subscriptions, in-app purchases, upgrade
screen, paid identity, purchase SDK, restore-purchase action, or subscription
terms. A paid model may be introduced only when a later release provides
substantive premium functionality.

## 3. Primary User Experience

### First Launch

The first launch shows no sample tasks. A maximum of three concise pages
explain:

1. Fast task capture.
2. Reminders are optional and permission is requested only when needed.
3. Tasks remain on the device and biometric protection is optional.

The introduction can be skipped. Completing or skipping it is persisted. The
user then enters an empty Inbox with a clear quick-add action.

### Core Navigation

On iPhone, the application retains four primary destinations: Today, Lists,
Focus, and Settings. On iPad, it uses adaptive navigation and constrained
content widths so that list and editor layouts do not appear as stretched
phone screens.

### Permission Timing

- Notification permission is requested only when the user first creates or
  enables a reminder.
- Biometric authentication is requested only when the user explicitly enables
  application lock.
- The application does not request location, calendar, contacts, camera,
  microphone, tracking, or photo-library access.
- A denied permission never prevents the underlying task from being saved.

## 4. Architecture

The existing Flutter application remains the implementation foundation.
Responsibilities currently concentrated in `AppStore` are separated into
testable units:

- `TaskRepository`: the authoritative interface for tasks, lists, and focus
  history.
- `SettingsRepository`: appearance, language, onboarding, biometric-lock, and
  other user preferences.
- `NotificationCoordinator`: notification permission state, scheduling,
  reconciliation, actions, and notification-driven navigation.
- `FocusSessionController`: active timer state, restoration, completion, and
  history creation.
- `AppController`: application-level coordination and observable UI state. It
  does not serialize data or directly invoke platform plugins.

Pages depend on these interfaces rather than plugin implementations. Platform
services are injected so unit and widget tests can use deterministic fakes.
Feature files remain focused; unrelated visual or architectural rewrites are
outside scope.

## 5. Data Model and Reliability

### SQLite Storage

Task, list, and focus data move from JSON strings in `SharedPreferences` to
SQLite. Preferences remain suitable only for small application settings.

The database provides:

- Explicit schema versioning.
- Transactions for multi-record changes.
- Foreign keys and indexes.
- Deterministic ordering fields.
- Startup integrity checks.
- Validation before writes.
- Clear handling of open, read, and write failures.

This release has no production users, so migration of existing prototype data
is not required. Existing sample JSON data is not imported.

### Data Rules

- The Inbox always exists and cannot be deleted.
- Deleting a custom list requires confirmation and an explicit choice to move
  its tasks to Inbox or delete them.
- Destructive bulk actions require confirmation.
- A failed write must not update the visible state as though it succeeded.
- A failed read or integrity check must not overwrite the original database.
- User-facing recovery offers retry and diagnostic export where applicable.
- Logs never include task titles, notes, subtask text, or list content.

### Time Representation

Persisted instants use UTC. User-selected local components and repeat rules are
stored with enough information to recreate the intended local schedule.
Display and notification scheduling convert using the device time zone.

The app reconciles future reminders after launch, foreground restoration,
relevant task edits, and detected time-zone changes.

## 6. Reminder Behavior

The reminder system supports one-time, daily, weekly, and monthly schedules.
Monthly reminders use the final valid day in months that lack the selected day.
For example, a reminder selected for the 31st occurs on February's final day.

Notification categories provide:

- Complete: marks the associated task complete and cancels future occurrences.
- Snooze 10 Minutes: schedules a one-time reminder 10 minutes from the action.
- Default open: opens the associated task.

Scheduling failures do not discard task changes. The affected task exposes a
clear retryable reminder state. Permission denial explains how to enable
notifications in Settings without repeatedly prompting.

Notification identifiers are stable across process launches and must not rely
on randomized runtime hash values. Reconciliation removes orphaned requests and
restores missing valid requests.

## 7. Focus Timer Behavior

Pomodoro, countdown, and count-up modes share one clock model. Elapsed and
remaining time derive from persisted start, pause, and target instants rather
than accumulated UI timer ticks.

An active session survives backgrounding and process termination. Countdown and
Pomodoro completion use a local notification; LessDo does not request a
background execution entitlement solely to keep a timer alive.

The user can pause, resume, cancel, or complete a session. Completion creates
one history record and optionally completes the bound task. Repeated lifecycle
events must not create duplicate history records.

## 8. Error Handling and Diagnostics

- Recoverable errors appear next to the action that failed.
- Long-running or async actions prevent accidental duplicate submission.
- UI code checks lifecycle state before applying async results.
- Storage, notification, biometric, parsing, and deep-link failures have
  distinct messages and recovery actions.
- Uncaught Flutter and platform errors are written to a bounded local
  diagnostic log.
- Diagnostic logs contain technical metadata only and are never uploaded
  automatically.
- Export occurs only after the user invokes the system share sheet.

## 9. Localization, Layout, and Accessibility

All user-visible strings are localized in English and Simplified Chinese.
Dates, times, numbers, and plural forms use the selected locale. The initial
language follows the system, and Settings offers System, English, and Simplified
Chinese.

The interface supports:

- iPhone and iPad size classes.
- Portrait and landscape orientations.
- System light and dark appearance.
- Dynamic Type without clipped primary actions.
- Keyboard and safe-area avoidance.
- Reduce Motion.
- VoiceOver labels, values, hints, and logical focus order.
- Minimum 44 by 44 point interaction targets.
- Status communication that does not rely on color alone.

Empty, denied-permission, unavailable-biometric, no-reminder, no-list, and
no-focus-history states each provide an explanation and a useful next action.

## 10. Privacy and App Store Compliance

The iOS target includes an application-owned `PrivacyInfo.xcprivacy` describing
the required-reason APIs used directly by LessDo. Third-party manifests are
verified during archive inspection.

The final privacy policy accurately states:

- Tasks, lists, preferences, and focus history are stored locally.
- LessDo 1.0 has no account, advertising SDK, analytics backend, or tracking.
- Notifications are used only for user-created reminders and focus completion.
- Biometric matching is performed by the operating system; biometric data is
  not received or stored by LessDo.
- List or diagnostic content leaves the device only when the user explicitly
  opens the system share sheet.

Before submission, the publisher must provide:

- A real support email.
- A public HTTPS privacy-policy URL.
- A support URL.
- App Store screenshots and localized listing text.
- Accurate App Privacy, age-rating, category, and content-rights answers.
- Signing certificates, provisioning, and App Store Connect access.

The release archive must contain no demo records, debug banners, placeholder
contacts, broken links, paid-product references, disabled future-feature
controls, or private APIs.

## 11. Testing Strategy

### Unit Tests

Unit tests cover:

- English and Chinese smart input, invalid input, noon/midnight, and explicit
  dates.
- One-time and repeat scheduling, month-end behavior, daylight-saving and time
  zone changes.
- Stable notification identifiers and reconciliation decisions.
- Task, list, subtask, and focus validation.
- Focus elapsed-time calculation, pause/resume, restoration, and idempotent
  completion.
- Deep-link validation and safe callback handling.

### Repository Tests

Tests use temporary SQLite databases and cover:

- Empty first launch.
- Transaction commit and rollback.
- Foreign-key and list deletion rules.
- Concurrent writes and deterministic ordering.
- Integrity failure behavior.
- Schema creation and supported upgrades.
- Preservation of the original database after a failed read or upgrade.

### Widget Tests

Widget tests cover:

- Introduction completion and skip.
- Empty Inbox and quick add.
- Task and list editing.
- Destructive confirmations.
- Notification denial and recovery guidance.
- Unsupported or failed biometric authentication.
- English and Simplified Chinese layouts.
- Large Dynamic Type and narrow/wide layouts.

### Integration and Manual Validation

Integration tests cover creating, editing, reminding, completing, restoring,
and deleting tasks; focus restoration; deep links; and persistence across
application restarts.

The release is built and exercised on current iPhone and iPad simulators.
Physical-device sign-off is required for Face ID or Touch ID, real notification
delivery, notification actions, rotation, and a signed TestFlight build.

## 12. Release Gates

LessDo 1.0 is release-ready only when all applicable gates pass:

- `flutter analyze` reports no issues.
- All unit, repository, widget, and integration tests pass.
- iPhone and iPad simulator builds and critical user journeys pass.
- An iOS Release build succeeds.
- Archive inspection verifies permissions, privacy manifests, identifiers,
  versioning, assets, and absence of forbidden placeholders.
- No known crash, data-loss defect, blocker accessibility issue, or critical
  functional defect remains.
- Localized privacy and review documents match the shipped binary.
- Physical-device checks and signed TestFlight validation pass.

Code and simulator work can prepare the release, but App Store submission
cannot be considered complete without publisher-owned signing credentials,
public support and privacy URLs, final store metadata, and physical-device
evidence.
